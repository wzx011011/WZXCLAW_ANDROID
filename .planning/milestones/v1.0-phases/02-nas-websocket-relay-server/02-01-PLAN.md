## Protocol Reference

The relay must transparently forward the wzxClaw desktop WebSocket protocol.
All messages are JSON: `{"event": "...", "data": ...}`.

Client events the relay will see:
- `ping` / `pong` -- application-level heartbeat, relay should NOT forward these
- `command:send` -- user command from mobile, forward to desktop
- `command:stop` -- stop generation, forward to desktop
- `connected` -- desktop confirms connection, forward to mobile
- `message:user` -- echoed user message, forward to mobile
- `message:assistant` -- full assistant message, forward to mobile
- `stream:text_delta` -- streaming text chunk, forward to mobile
- `stream:tool_use_start` -- tool use badge, forward to mobile
- `stream:done` -- generation complete, forward to mobile
- `stream:error` -- error during streaming, forward to mobile
- `session:messages` -- history sync, forward to mobile

Desktop connects to relay URL: `ws://relay-host:port/?token=XXX&role=desktop`
Mobile connects to relay URL: `ws://relay-host:port/?token=XXX&role=mobile`
(or mobile is the default role if no role param is provided)

  <behavior>
    - Server starts on configurable PORT (default 8080)
    - WebSocket connection without token query param is rejected with code 4001 and reason "missing token"
    - WebSocket connection with wrong token is rejected with code 4001 and reason "invalid token" (for v1: token matches env AUTH_TOKEN, a simple shared secret)
    - Desktop connects with `?token=XXX&role=desktop`, mobile connects with `?token=XXX&role=mobile`
    - If role is missing, default to "mobile"
    - When both desktop and mobile with same token are connected, they form a "room"
    - Messages from desktop (except ping/pong) are forwarded to the paired mobile
    - Messages from mobile (except ping/pong) are forwarded to the paired desktop
    - ping/pong messages are consumed by the relay (logged, not forwarded)
    - When desktop disconnects, mobile receives a system message: {"event":"system:desktop_disconnected"}
    - When mobile disconnects, desktop receives a system message: {"event":"system:mobile_disconnected"}
    - When a client disconnects, its room entry is cleaned up
    - If a second desktop connects with the same token, the first desktop is disconnected (code 4002, "replaced")
    - Log all connections, disconnections, room joins, and message forwarding with timestamps
  </behavior>

    **relay/lib/logger.js:**
    Export three functions: `log(msg)`, `warn(msg)`, `error(msg)`.
    Each prefixes with ISO timestamp and category tag: `[RELAY YYYY-MM-DDTHH:mm:ss.SSSZ] [INFO|WARN|ERROR] msg`.
    Use `process.stdout.write` for log, `console.error` for warn and error.

    **relay/lib/auth.js:**
    Export a single function `authenticate(token)` that returns `true` if:
    - `token` is a non-empty string AND
    - `token` equals `process.env.AUTH_TOKEN`
    - If AUTH_TOKEN env var is not set, log a warning ONCE at startup and accept any non-empty token (dev mode).
    The function should return `{ ok: boolean, reason: string }`.

    **relay/lib/room.js:**
    Export a `RoomManager` class that manages rooms keyed by token string.

    Internal structure: `Map<string, { desktop: WebSocket|null, mobile: WebSocket|null }>`

    Methods:
    - `join(token, role, ws)` -- Add ws to the room under the given role ("desktop" or "mobile"). If role is "desktop" and a desktop already exists for this token, close the old one with code 4002 and reason "replaced by new connection". Set up ws event handlers for message forwarding and cleanup.
    - `_forward(from, to, data)` -- Send data to the `to` WebSocket if it exists and is open (readyState === 1). Silently skip if to is null or closed.
    - `_onMessage(token, role, ws, data)` -- Parse data as JSON. If event is "ping" or "pong", log and return (do not forward). Otherwise forward to the other side: if role is "desktop", forward to mobile; if role is "mobile", forward to desktop. If JSON parse fails, log a warning and skip.
    - `_onDisconnect(token, role, ws)` -- Remove the client from its room slot. Send system notification to the other side ({"event":"system:desktop_disconnected"} or {"event":"system:mobile_disconnected"}). If both sides are null, delete the room entirely. Log the disconnection.
    - `_sendSystem(ws, event)` -- Helper to send `JSON.stringify({event})` if ws is open.
    - `getRoomCount()` -- Return number of active rooms (for health checks).

    Important: Use `ws.on('message', ...)` and `ws.on('close', ...)` to wire up forwarding and cleanup. Store cleanup handlers so rooms are always cleaned up.

    **relay/server.js:**
    1. Import `ws` (WebSocketServer from 'ws'), auth, room, and logger.
    2. Read PORT from env (default 8080).
    3. Read AUTH_TOKEN from env. If not set, log warning: "AUTH_TOKEN not set - accepting any token (dev mode)".
    4. Create WebSocketServer on PORT.
    5. On `connection`, parse the URL query params for `token` and `role`.
       - If token is missing or authenticate returns false: close ws with code 4001 and the reason, log the rejection.
       - If authenticated: log the connection, call `roomManager.join(token, role, ws)`.
    6. Log "Relay server listening on port {PORT}" on startup.
    7. Handle process SIGTERM and SIGINT for graceful shutdown (close all connections, then exit).
    8. Periodically log room count every 60 seconds (setInterval).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> relay | Untrusted input from any network client. Token in query string, JSON messages in WebSocket frames. |
| relay -> relay | Internal room management. Trusted internal state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | S (Spoofing) | WebSocket connection | mitigate | Token authentication required on every connection. Invalid token = immediate rejection with 4001 close code. |
| T-02-02 | T (Tampering) | Message forwarding | accept | Relay does NOT modify message payloads -- transparent forwarding. Integrity is end-to-end between desktop and mobile. |
| T-02-03 | R (Repudiation) | Connection logging | mitigate | All connections, disconnections, and auth failures are logged with timestamps. |
| T-02-04 | I (Info Disclosure) | Token in URL query string | mitigate | Token is passed in query string which may appear in logs. In production (HTTPS via nginx), query string is encrypted in TLS. Document that HTTPS is mandatory. |
| T-02-05 | D (DoS) | WebSocket connections | accept | Personal tool, single user. No rate limiting needed. If exposed publicly, rely on nginx rate limiting. |
| T-02-06 | E (Elevation) | Room management | mitigate | Token isolates rooms. Clients with different tokens cannot see each other's messages. Desktop replacement (4002) prevents stale ghost connections. |
</threat_model>
